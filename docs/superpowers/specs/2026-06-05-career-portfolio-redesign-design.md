# 이직용 포트폴리오 개편 설계

## 세 줄 요약

- 승인된 방향은 A안인 "경력 증거 먼저" 구조이며, AI 협업 프로젝트는 전면 대표가 아니라 개발 프로세스 개선 증거로 배치한다.
- 첫 화면은 지원 포지션, 핵심 도메인, 연락 CTA를 명확히 하고, 최신 경력과 `castleCms`를 클릭 없이 스캔할 수 있게 만든다.
- 구현 범위는 Hero/CTA, 경력 상세 강화, 대표 프로젝트 상세화, AI 협업 사례, Build Log/대표 글, 데스크탑 UI 정리까지 1~6번 순서를 포함한다.

## 승인된 접근

사용자는 브라우저 companion에서 A/B/C 방향을 비교한 뒤 A안을 승인했다.

승인된 A안:

```text
Hero / CTA
-> Hiring Snapshot
-> 소프트위즈 중심 경력 상세
-> castleCms 대표 프로젝트 상세
-> AI 협업 프로세스 증거
-> Build Log / 대표 글
-> Desktop UI polish
```

이 순서는 "AI를 사용했다"보다 "운영/결제/관리자 도구를 다뤄본 풀스택 개발자"라는 증거를 먼저 보여준다.

## 목표

이 개편의 목표는 채용자 또는 기술 리드가 짧은 시간 안에 아래 질문에 답할 수 있게 만드는 것이다.

- 어떤 포지션에 맞는 개발자인가?
- 최근 실무에서 어떤 도메인과 문제를 다뤘는가?
- 개인 프로젝트가 실무 경험과 어떻게 이어지는가?
- AI 도구를 사용했다면 구현 대체가 아니라 어떤 개발 프로세스를 개선했는가?
- 연락하거나 이력서를 요청할 수 있는 경로가 명확한가?

## 범위

### 1. Hero + Resume CTA

첫 화면의 문구를 더 채용 중심으로 조정한다.

추천 headline:

```text
운영/결제/관리자 도구 경험이 있는 풀스택 개발자 김보성입니다.
```

추천 보조 문구:

```text
Go/Node.js/Spring 기반 API, React/Next.js 화면, SQL 운영 데이터를 함께 다루며
결제/정산/CRM/관리자 도구의 기능 설계와 배포 검증을 경험했습니다.
```

CTA는 다음 세 가지를 둔다.

- `이력서 요청`: 현재 공개 이력서 파일이 없으므로 `mailto:` 기반으로 둔다.
- `GitHub`: 기존 GitHub 링크를 유지한다.
- `Mail`: 기존 메일 링크를 유지한다.

향후 `public/resume.pdf`가 추가되면 `이력서 요청`을 `이력서 보기` 링크로 바꿀 수 있다.

### 2. 경력 상세 강화

경력은 최신성과 실무 증거가 가장 중요하므로 소프트위즈, 링크커넥션, 일인치를 우선 강화한다.

리스트 화면에서 바로 보일 요약 필드를 추가한다.

- `summary`: 한 문장 요약
- `domain`: 결제/정산/CRM/병원 운영/React Native 앱 등 도메인
- `techStack`: 주요 기술
- `featured`: 최신 핵심 경력 여부
- `topHighlights`: 채용자가 바로 볼 2~3개 핵심 증거

마크다운 상세는 유지하되, 화면 상단에는 JSON 기반 요약과 핵심 증거를 먼저 노출한다.

소프트위즈에서 강조할 내용:

- STICPAY E-Wallet 신규 결제수단 도입
- Cellxpert 파트너 정산 수동 실행 안정화
- 계좌/거래/문서 소유권 검증, 2FA/비밀번호 재설정 등 보안 보강
- Gateway DEV 배포 자동화와 SHA 기반 런타임 검증

링크커넥션에서 강조할 내용:

- 병원 운영 솔루션 화면과 API 연동
- React 기반 사용자 플로우와 반응형 UI 구현
- Node.js 기반 백엔드 기능의 Spring Boot 구조 마이그레이션
- GitHub Actions, CloudFront, Slack Webhook 기반 배포 운영

일인치에서 강조할 내용:

- WebView 운영 앱을 React Native 앱 구조로 재구현
- 기존 소스 접근이 제한된 상황에서 화면/기능 동작을 분석해 재구성
- 게시글, 이미지 업로드, 소셜 로그인, 지도 WebView, 딥링크, 결제 기능 구현
- Google Play Console과 Apple App Center 배포/테스트 대응

### 3. `castleCms` 대표 프로젝트 상세화

`castleCms`는 "기능이 많은 개인 프로젝트"보다 "운영형 CMS 설계/검증 사례"로 보이게 한다.

프로젝트 데이터 모델에 아래 의미를 분리한다.

- `role`: 본인 역할
- `problem`: 해결하려 한 운영 문제
- `solution`: 구조와 구현 방향
- `result`: 현재 검증된 결과 또는 다음 검증 기준
- `aiUsage`: AI 도구 활용 방식
- `repoVisibility`: 공개 여부
- `reviewAvailable`: 필요 시 코드 열람 가능 여부

화면에서는 기존 스크린샷 갤러리를 유지하되, 좌측 설명은 다음 순서로 정리한다.

```text
문제
내 역할
설계/구현
검증
AI 활용
```

성과 숫자를 만들지 않는다. 아직 수치가 없는 항목은 "검증 기준"과 "다음 개선 계획"으로 표현한다.

### 4. AI 협업 프로젝트 추가

`AI-assisted Portfolio Rebuild`는 대표 프로젝트 첫 번째로 세우지 않는다. 대신 개발 프로세스 개선 사례로 추가한다.

안전한 제목:

```text
AI 협업 기반 포트폴리오 개선
```

핵심 메시지:

```text
Codex/Cursor를 활용해 요구사항 정리, UI 개선, 이미지 리소스 관리,
분석 이벤트 전환, 핸드오프 문서화, 배포 검증을 반복한 프로젝트입니다.
```

이 섹션은 "AI가 만들어준 프로젝트"가 아니라 "AI와 함께 일하는 방식의 증거"로 읽히게 한다.

### 5. Build Log / 대표 글

짧은 작업 기록과 대표 글을 홈 하단에 둔다.

Build Log 후보:

- softwiz 로고 리소스 정리와 표시 비율 개선
- Vercel Analytics 유료 제한 확인 후 GA4/GTM 이벤트 구조로 전환
- 연락/내비게이션/블로그/프로젝트 클릭 이벤트 설계
- 핸드오프 문서 작성과 커밋 단위 기록
- 공개 URL 기준 QA와 배포 검증

대표 글은 블로그 전체 목록을 보여주기보다 3~5개만 선별한다. 초기 구현은 빌드 안정성을 위해 정적 추천 목록으로 둔다. 이후 블로그 데이터 소스가 안정적으로 통합되면 자동 추천으로 바꿀 수 있다.

### 6. Desktop UI polish

데스크탑에서는 탭을 눌러야만 프로젝트가 보이는 구조를 줄인다.

권장 화면 구조:

- 데스크탑 홈에서는 `Projects`와 `WorkHistory`를 모두 노출한다.
- `ProfileTabs`는 모바일 압축 UI로만 유지한다. 데스크탑에서는 탭을 숨기고 프로젝트와 경력을 순서대로 렌더링한다.
- 랜덤 명언 바는 이직용 첫인상에서 정보 밀도가 낮으므로 제거하거나 작게 낮춘다.
- 섹션 내부 제목은 전역 `h2/h3/h4` 크기에만 의존하지 말고 로컬 Tailwind 클래스로 밀도를 조절한다.

## 컴포넌트 설계

예상 변경 컴포넌트:

- `src/components/Information/index.tsx`
  - headline, proof points, CTA를 조정한다.

- `src/components/WorkHistory/data.json`
  - 요약/도메인/기술스택/핵심 증거 필드를 추가한다.

- `src/components/WorkHistory/typs.ts`
  - 새 경력 데이터 필드를 반영한다.

- `src/components/WorkHistory/components/WorkHistoryItem.tsx`
  - 회사 메타, 요약, 핵심 증거, 기술 스택을 먼저 보여주고 마크다운 상세를 이어서 보여준다.

- `src/components/Projects/data.json`
  - `castleCms` 데이터 구조를 문제/역할/해결/검증/AI 활용 중심으로 확장한다.
  - AI 협업 기반 포트폴리오 개선 프로젝트를 보조 프로젝트로 추가한다.

- `src/components/Projects/types.ts`
  - 확장된 프로젝트 필드를 타입에 반영한다.

- `src/components/Projects/components/ProjectItem.tsx`
  - 새 필드가 있으면 구조화된 섹션으로 보여주고, 기존 필드도 호환한다.

- `src/app/page.tsx`
  - 데스크탑에서 경력과 프로젝트가 탭 안에 숨지 않도록 섹션 순서를 조정한다.

새 컴포넌트는 아래 범위에서 추가한다.

- `src/components/BuildLog`
- `src/components/FeaturedWriting`

`CareerSummary`는 `Information` 내부에서 먼저 해결하고, 중복이 커질 때만 분리한다.

## 데이터 흐름

- 홈 페이지는 정적 JSON과 기존 마크다운 경력 문서를 함께 사용한다.
- WorkHistory 리스트는 JSON 요약을 먼저 렌더링하고, 상세 설명은 기존 마크다운을 그대로 읽는다.
- 프로젝트 섹션은 JSON 데이터만으로 렌더링한다.
- 분석 이벤트는 기존 `trackEvent` 헬퍼를 유지하며 새 CTA와 프로젝트 링크에도 필요한 경우 추가한다.

## 오류와 빈 상태

- 이력서 파일이 없으므로 초기 구현은 `mailto:` CTA로 처리한다.
- 프로젝트에 스크린샷이 없는 경우 이미지 영역을 렌더링하지 않는다.
- 확장 필드가 없는 오래된 프로젝트 데이터는 기존 `summary`, `description`, `highlights`로 표시한다.
- 오래된 경력은 `topHighlights`가 없어도 기존 마크다운 상세로 표시되게 한다.

## 테스트와 QA

필수 검증:

- `tsc --noEmit`
- `next build`
- 기존 Jest 테스트가 있으면 관련 테스트 갱신 후 실행

브라우저 QA:

- 1440x900에서 첫 화면에 headline, CTA, Hiring Snapshot이 보이는지 확인
- 데스크탑에서 프로젝트와 경력이 탭 클릭 없이 스캔되는지 확인
- 390x844 모바일에서 섹션이 과하게 길어지거나 텍스트가 겹치지 않는지 확인
- 다크모드에서 카드, 칩, 버튼 대비가 충분한지 확인
- 프로젝트 스크린샷 갤러리와 원본 이미지 열기가 유지되는지 확인

## 제외 사항

- 실무 성과 숫자를 임의로 만들지 않는다.
- 공개 이력서 PDF가 준비되지 않은 상태에서 가짜 다운로드 버튼을 만들지 않는다.
- AI 협업 프로젝트를 첫 번째 대표 프로젝트로 세우지 않는다.
- 전체 디자인 시스템을 새로 만들지 않는다.
