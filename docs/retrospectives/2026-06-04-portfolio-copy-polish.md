# 포트폴리오 문구 개선 배포 회고

## 세 줄 요약

- 첫 화면과 castleCms 프로젝트 설명을 채용담당자 관점에서 더 명확하게 다듬어 main에 배포했다.
- 블로그 API 진입점을 facade 구조로 정리해 이후 GitHub Markdown 글 연동 준비를 마쳤다.
- 테스트, 타입체크, 빌드, 공개 URL HTML 확인, 홈/블로그/다크모드 캡처 QA를 완료했다.

## 배포 내용

이번 배포는 포트폴리오의 첫인상과 개인 프로젝트 설명력을 높이는 데 집중했다. 첫 화면의 H1은 “서비스를 끝까지 다루는 풀스택 개발자 김보성”으로 정리했고, 소개 문장은 웹/앱, 백엔드, 배포/운영, AI 개발 도구 활용이 함께 드러나도록 수정했다.

castleCms 프로젝트는 단순 개인 프로젝트 소개가 아니라 운영형 관리자 도구를 설계하고 검증한 프로젝트로 보이도록 정리했다. 요구사항 정의, 화면 구조, API 계약, 인증/권한, 데이터 모델, 테스트 기준을 먼저 잡고 구현했다는 흐름을 설명에 반영했다.

블로그 쪽은 현재 Notion을 사용하지만, UI가 Notion 구현을 직접 참조하지 않도록 `src/app/(blogLayout)/api/blog.ts`를 활성 블로그 소스를 감싸는 facade 형태로 정리했다. 이후 GitHub Markdown 글을 추가할 때 UI 페이지를 크게 수정하지 않고 `api/blog.ts` 안에서 출처를 확장할 수 있다.

## 검증 결과

- `npm.cmd test -- --runInBand`
  - 6개 테스트 스위트 통과
  - 8개 테스트 통과
  - 스냅샷 1개 통과

- `npm.cmd run build`
  - Next.js production build 통과
  - 정적 페이지 생성 10개 완료

- `npm.cmd exec tsc -- --noEmit`
  - 단독 실행 기준 통과
  - `next build`와 동시에 실행하면 `.next/types` 생성 타이밍이 겹쳐 실패할 수 있어, build 완료 후 단독 실행으로 확인했다.

- 공개 URL 확인
  - `https://bocelog.vercel.app/` 응답 200
  - 홈 문구 `서비스를 끝까지 다루는` 반영 확인
  - castleCms 설명 `운영형 관리자 도구의 핵심 흐름` 반영 확인
  - `https://bocelog.vercel.app/blog/1ac5abe4-94ef-80c5-81cb-c236d3d60daa` 응답 200
  - 블로그 상세 제목과 목차 렌더링 확인

## 화면 QA

Chrome headless 캡처로 다음 화면을 확인했다.

- 홈 데스크톱
  - 첫 화면 문구가 반영됐고 주요 정보가 첫 viewport 안에서 읽힌다.

- 홈 모바일
  - 이전에 보였던 오른쪽 잘림이 사라졌다.
  - H1과 소개 문단이 390px 폭에서도 읽힌다.

- 홈 다크모드
  - 배경과 텍스트 대비가 충분하고 링크와 탭 상태가 식별된다.

- 블로그 상세 데스크톱
  - 제목, 본문 이미지, 우측 목차가 렌더링된다.

## 문제와 대응

로컬 개발 환경에서는 Notion API token이 유효하지 않으면 블로그 상세가 500으로 보인다. 이 문제는 로컬 환경 변수 문제이며, 공개 Vercel 환경에서는 블로그 상세 URL이 200으로 응답하고 목차도 렌더링되는 것을 확인했다.

검증 중 `npm.cmd exec tsc -- --noEmit`와 `npm.cmd run build`를 병렬로 실행하면 `.next/types` 파일 생성 타이밍이 겹쳐 타입체크가 실패했다. 이후 build 완료 후 타입체크를 단독 실행해 통과를 확인했다. 앞으로 Next.js 타입체크는 build와 동시에 돌리지 않고 순차 실행하는 편이 안전하다.

## 다음 개선 후보

다음 작업은 GitHub Markdown 글 소스를 실제로 붙이는 것이다. 이미 `api/blog.ts`에 facade 경계를 만들었으므로, 다음 단계에서는 GitHub 글 목록을 읽는 소스 모듈을 추가하고 Notion 글과 병합 또는 별도 탭으로 노출하는 방식을 선택하면 된다.

또한 개인 프로젝트 탭은 테스트로 전환 동작을 확인했지만, 브라우저 자동 클릭 기반 시각 QA까지 더하면 이미지 갤러리와 상세 이미지 새 창 열림까지 더 강하게 검증할 수 있다.
