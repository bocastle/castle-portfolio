# 블로그 목차 배포 회고

## 세 줄 요약

- 블로그 상세 글의 `##`, `###` 제목을 자동 목차로 만들고, 클릭 시 부드럽게 해당 위치로 이동하도록 개선했다.
- 처음에는 본문 상단에 목차를 배치했지만 글 흐름을 끊는 문제가 있어 데스크톱에서는 우측 sticky 사이드바로 이동했다.
- 테스트와 타입체크는 통과했으며, 로컬 브라우저 검증은 Notion 글 데이터와 브라우저 연결 제약으로 일부 제한이 있었다.

## 배포 브랜치와 커밋

- 작업 시작 브랜치: `dev`
- 기능 브랜치: `feature/blog-toc`, `feature/blog-toc-sidebar`
- 병합 대상: `dev`, `main`
- 주요 커밋:
  - `579ba85 블로그 자동 목차 추가`
  - `8a6615a 블로그 목차 사이드바 배치`

## 배포 목적

블로그 상세 글에서 독자가 긴 글의 구조를 빠르게 파악하고 원하는 섹션으로 이동할 수 있도록 목차 기능을 추가했다. 특히 기술 글은 이미지, 코드블록, 여러 소제목이 섞여 있어 본문 흐름을 따라가기 어렵기 때문에, 글 우측의 빈 공간을 활용해 탐색성을 높이는 방향으로 개선했다.

## 주요 변경 사항

- 마크다운 본문에서 `##`, `###` 제목을 추출해 목차 항목으로 변환했다.
- 코드블록 안의 heading 문법은 목차에서 제외했다.
- 중복 제목이 있을 경우 `-1`, `-2` 형태로 id가 겹치지 않도록 처리했다.
- 렌더링되는 `h2`, `h3`에도 목차와 같은 slug 규칙으로 id를 부여했다.
- 전역 CSS에 `scroll-behavior: smooth`를 적용해 목차 클릭 시 부드럽게 이동하도록 했다.
- 제목에 `scroll-mt-24`를 적용해 이동 후 제목이 화면 상단에 너무 붙지 않도록 했다.
- 데스크톱 `xl` 이상에서는 목차를 본문 우측 sticky 사이드바로 배치했다.
- 모바일과 태블릿에서는 우측 공간이 부족하므로 본문 상단에 목차를 유지했다.
- 배포 후 화면 확인 과정에서 sticky 고정 동작과 absolute 배치가 글 흐름과 맞지 않는다는 피드백을 반영했다. 이후 목차가 콘텐츠 영역 아래에서 시작해 고정처럼 보이는 문제가 남아, 글 상세 전체 레이아웃의 우측 컬럼으로 목차를 이동했다.

## QA 및 검증 결과

- `npm.cmd exec tsc -- --noEmit` 통과
- `npm.cmd test -- --runInBand __tests__/blog-toc.test.ts` 통과
- TOC 유틸 테스트 항목:
  - `##`, `###` heading 추출
  - 코드블록 내부 heading 제외
  - 중복 제목 slug 처리
- `git diff --check` 통과
- `npm run build`는 로컬 환경에서 Next 초기 출력 이후 장시간 진행되지 않아 제한 시간 안에 완료 확인하지 못했다.

## 확인한 화면 동작

로컬 `/blog` 페이지는 `200`으로 정상 응답했다. 다만 로컬 환경에서는 상세 글 링크가 서버 HTML에 내려오지 않아 실제 상세 글의 목차 렌더링을 완전히 확인하지 못했다. 배포 환경에서는 사용자가 실제 글 상세 화면에서 본문 상단 목차가 표시되는 것을 확인했고, 이후 데스크톱 기준 우측 빈 공간에 목차를 배치하는 방식으로 수정했다.

## 발생한 문제

- 처음 목차를 본문 상단에 배치했더니 이미지와 본문 사이에 큰 박스가 들어가 글 흐름을 끊었다.
- 이후 우측 목차를 sticky와 absolute 조합으로 배치했지만, 사용자가 기대한 동작은 화면 고정처럼 보이는 목차가 아니라 본문 레이아웃 안에서 함께 이동하는 보조 목차였다.
- sticky와 absolute를 제거한 뒤에도 목차가 `PageContent` 내부에서 시작해 대표 이미지 아래에 배치되면서, 스크롤 중 화면 상단에 걸려 고정처럼 보이는 문제가 남았다.
- 내장 브라우저 연결이 `windows sandbox failed: spawn setup refresh` 문제로 실패해 자동 브라우저 검증을 진행하지 못했다.
- Jest 실행 중 커버리지 파일 쓰기에서 `EPERM` 경고가 발생했다. 테스트 자체는 통과했으며, 샌드박스의 `coverage` 파일 쓰기 제한으로 판단했다.
- `tsc` 실행 중 `tsconfig.tsbuildinfo` 쓰기 권한 문제가 발생해 권한 밖 실행으로 타입체크를 재확인했다.

## 해결 방법

- 목차 컴포넌트에 `inline`, `sidebar` variant를 추가해 화면 폭에 따라 다른 배치를 사용하도록 했다.
- 데스크톱에서는 `absolute left-full` 배치를 제거하고, 글 상세 페이지 최상위 레이아웃을 `grid-cols-[minmax(0,56rem)_16rem]`로 구성해 제목, 대표 이미지, 본문, 이전/다음 글 영역 전체 옆에 목차를 배치했다.
- 모바일에서는 `xl:hidden`으로 기존 본문 상단 목차를 유지했다.
- 긴 목차는 `max-h-[calc(100vh-8rem)]`와 `overflow-y-auto`로 화면을 넘치지 않게 했다.
- 타입체크와 테스트는 권한 제약을 분리해서 다시 실행했고, 기능 검증 결과를 기준으로 배포했다.

## 배운 점

목차는 기능 자체보다 배치가 더 중요했다. 본문 위에 목차를 두면 구현은 단순하지만, 이미지와 글의 흐름을 끊어 실제 읽기 경험이 어색해질 수 있다. 블로그 상세 화면처럼 본문 폭이 좁고 우측 공간이 남는 구조에서는 sticky 사이드바가 더 자연스럽다.

또한 배포 전 로컬 브라우저 검증이 막힐 때는 테스트, 타입체크, HTTP 응답, 사용자의 배포 화면 확인을 나누어 검증 근거를 정리해야 한다. 자동화된 검증과 실제 화면 확인은 서로 대체가 아니라 보완 관계로 보는 것이 맞다.
