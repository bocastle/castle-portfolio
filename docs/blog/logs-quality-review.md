# GitHub logs 공개 글 품질 검수

## 검수 기준

`bocastle/logs` 글은 포트폴리오 블로그에 함께 노출되므로, 공개 메타데이터는 아래 기준을 통과해야 한다.

- 제목은 깨진 문자가 없어야 한다.
- 설명은 30자 이상 120자 이하로, 글의 실무 의미를 바로 설명해야 한다.
- slug는 `/blog/logs-...` 형식의 영문 소문자 URL이어야 한다.
- 카테고리와 태그는 목록/상세/필터 UI에서 읽기 쉬운 표시명으로 보여야 한다.
- 홈과 블로그 상단 글 묶음은 운영/배포, 장애 대응, 성능, 캐시, 테스트 흐름을 자연스럽게 읽히도록 구성한다.
- 상단 글은 제목, 설명뿐 아니라 본문 도입부와 섹션명도 개발 기록 톤으로 읽혀야 한다.

## 2026-06-05 본문 톤 점검

상단 글 5개는 `bocastle/logs` 원본 Markdown을 직접 수정했다. 포트폴리오 쪽에서 본문을 덮어쓰지 않고, GitHub raw Markdown이 그대로 렌더링되는 구조를 유지한다.

| 글 | 수정 방향 |
| --- | --- |
| CI/CD 파이프라인 운영 흐름 | 용어 정의보다 배포 검증 흐름과 롤백 기준을 먼저 설명하도록 도입부와 섹션명을 정리 |
| JPA N+1 쿼리 성능 개선 | 면접 답변식 정의를 줄이고, 느린 목록 API를 의심하는 흐름으로 도입부를 변경 |
| 외부 서비스 장애 대응 전략 | 종합 정리 문구를 덜고, 외부 API 실패 경계를 운영 관점에서 설명하도록 수정 |
| Spring Boot와 Next.js 캐시 전략 | Spring/Next.js 캐시 차이를 계층별 운영 기준으로 읽히도록 도입부를 추가 |
| 테스트하기 쉬운 코드의 조건 | 이모지 heading과 교재식 정의를 제거하고, 변경 검증 관점의 글로 정리 |
